# 知知 — 产品需求文档

## 一、一句话定义

管理者发出任何指令（语音、文字、图片），大模型理解后交给 Agent Teams 执行，管理者只管发令和拍板。

## 二、产品背景

### 2.1 核心矛盾

管理者的价值在于判断和决策，但他们的时间大量消耗在信息搬运、协调催办、系统操作、进度追踪上。

### 2.2 现有工具为什么没解决

- CRM、OA、ERP 是为执行动作设计的，不是为管理行为设计的
- IM 是为沟通设计的，管理动作淹没在消息流里
- AI 助手是为问答设计的，能回答但不能执行
- 自动化工具是为预定义流程设计的，不支持开放式指令

### 2.3 AI 时代的机会

管理者真正需要的是：一支随时待命的团队，我说什么他们就去做什么，做的过程中我能看见，关键节点来问我，最后给我结果。

这支团队现在可以用大模型 + Agent Teams 来实现。

### 2.4 产品定位

不做 CRM/OA 插件，不做聊天助手，不做语音版企业 IM。

做 AI 时代原生的管理者控制台 — 管理者和一支 AI Agent Team 之间的指挥界面。

## 三、AI 时代的管理哲学

本产品基于一个核心判断：AI 时代，管理方式会发生根本变化。

### 3.1 管理者角色的变化

过去管理者同时承担"推动者"和"决策者"两个角色。AI 时代，推动者角色会被 AI 大幅接管，管理者回归真正的管理：

- **目标设计者**：定义要达成什么结果，而不是拆解每一步操作
- **边界设计者**：定义哪些事 AI 可以自动做，哪些必须先问人
- **例外裁决者**：只在异常、冲突、高风险节点介入
- **责任承担者**：对最终组织结果负责

### 3.2 产品第一性原理

**默认 AI 推进，人只在关键节点出手。**

这意味着：

- AI 不等管理者发问，而是持续推进
- 管理者不需要盯全部过程，只盯例外
- 大部分时候管理者不在产品里，产品自己在跑
- 管理者的时间应该花在判断上，不是操作上

### 3.3 核心管理对象

产品不预定义事项类型或管理类别。管理者发出的任何指令，大模型自行理解和分类。

但从管理本质看，管理者表达的意图通常属于以下几类：

- **目标**：持续运行的结果要求（"本周把客户满意度提上去"）
- **关注**：持续运行的监控（"帮我盯着华东区异常"）
- **规则**：持续生效的管理约束（"涉及预算的事一律先问我"）
- **事项**：具体的一次性管理动作（"把这个客户交给小王"）

这些不需要管理者手动分类，大模型自动识别。

## 四、目标用户

### 4.1 核心画像

任何需要"对结果负责、通过调度他人和资源来达成目标"的人。

共同特征：

- 管的事多，切换快，时间碎片化
- 不想亲自操作系统，想直接说结果要求
- 愿意把执行交给 AI，但必须保留拍板权
- 对"说了之后有没有人跟进"有强烈焦虑

### 4.2 典型角色

不按行业限定，按管理行为定义：

- 小团队负责人
- 创业公司老板
- 项目负责人
- 区域负责人
- 门店负责人
- 运营/客服/销售管理者

### 4.3 第一批最适合的用户

高频调度型管理者 — 每天都在说"帮我查一下""这个谁跟一下""今天给我结果""催一下负责人"的人。

### 4.4 MVP 阶段不优先服务

- 需要精细手工录入的执行岗
- 只想和 AI 聊天的个人用户
- 对 AI 执行完全不信任、什么都要手动操作的人
- 强依赖严密审批合规链条的场景

## 五、会话模型

### 5.1 多会话结构

知知采用多会话模型，类似 ChatGPT 的会话结构：

- 每个会话是一个独立的上下文容器
- 会话内可以发多条指令、产生多个任务
- 不同会话之间上下文隔离
- 用户通过新建会话来开启新主题
- 多个会话可以同时有任务在执行，实现多任务并行

### 5.2 会话与任务的关系

- 一个会话包含多条消息（用户指令 + AI 响应）
- 每条用户指令可能产生一个任务
- 任务的执行状态在会话内实时更新
- 会话是管理者和 AI 参谋之间的对话线程

### 5.3 Local-First 架构

数据先存本地，再同步到服务端：

- 所有数据先写入本地 SQLite 数据库
- UI 从本地读取，秒级响应
- 后台定期同步到服务端
- 离线时正常使用，网络恢复后自动同步
- 服务端推送实时写入本地

## 六、核心架构

整个产品只有 3 层。

### 6.1 输入层

管理者可以用任何方式发出指令：

- **语音**：按住说话，松开发出
- **文字**：打字输入
- **图片**：拍照、截图、相册选择
- **混合**：语音 + 图片、文字 + 图片

不限格式，不限内容，不限长度。管理者怎么跟真人助手说话，就怎么跟这个产品说话。

### 6.2 理解层

大模型接收多模态输入后，结合管理者上下文进行理解：

- 管理者想要什么结果
- 需要做哪些事
- 有没有风险或需要确认的地方
- 是否关联之前的指令或正在进行的任务

理解层的输出不是一段文字回复，而是一个可执行的任务描述，交给 Agent Teams。

理解层依赖的上下文：

- 管理者身份信息（角色、职责）
- 历史指令和执行记录
- 当前正在进行的任务
- 管理者设定的偏好和规则

### 6.3 执行层（Agent Teams）

Claude Agent Teams 接收任务后开始执行：

- 主 Agent（参谋长）拆解任务，决定需要几个子 Agent、分别做什么
- 子 Agent 各自调用工具执行
- 可以并行、可以串行、可以嵌套
- 遇到需要管理者确认的节点，暂停并请示
- 执行完成后汇总结果返回

Agent Teams 的能力由其可用的工具决定。工具越多，能做的事越多。

## 六、多任务并行

管理者的工作不是"一次只办一件事"，而是连续发令、随时插队、临时拍板。多任务并行是产品的基础能力，不是后续增强。

### 6.1 多任务模型

同一时间允许多个任务并行执行。任务按管理者关注度分为 4 层：

- **待拍板**：最高优先级，任何需要确认的任务。始终置顶。
- **执行中**：AI 正在推进的任务。后台并行跑，首页展示摘要。
- **持续关注**：长期任务（"帮我盯着""每天 9 点给我"），不占前台注意力。
- **已完成 / 失败**：历史沉淀层，用于回看和复盘。

### 6.2 局部阻塞

某个任务进入"待确认"不影响其他任务继续执行。确认机制是局部的，不是全局的。

### 6.3 任务归属识别

用户发一句话时，系统必须判断这是新任务还是旧任务的追问/补充。

- "刚才那个，加一个条件" → 挂到旧任务
- "另外，你帮我查一下华东区销量" → 新建任务
- 置信度低时主动反问："这句话是关于刚才的客户任务，还是新建一个任务？"

### 6.4 优先级排序

任务流的排序由系统承担，不让用户手动管理列表。排序综合考虑：

- 是否待确认
- 是否异常或失败
- 是否超时
- 是否用户刚刚交代
- 是否高影响任务

用户也可以通过一句话调整："这个先做""这个放后面""这个今天一定要出结果"。

## 七、产品界面

### 7.1 设计原则

- 极简，不做复杂导航和菜单
- 管理者打开产品，要么发指令，要么看进展
- 所有信息围绕"指令"组织，不围绕"功能模块"组织
- 首页不是聊天历史，而是"当前战况"

### 7.2 界面结构

整个产品只有 3 个核心视图。

#### A. 首页（发令台 + 战况）

首页同时承担发令和看战况两个职责。

顶部区域：**待拍板**（0—3 条最紧急确认项）

主区域：**任务流**（按分层排列：执行中 → 持续关注 → 已完成）

- 每条任务只露出一句最关键进展
- 执行中任务很多时自动聚合为摘要（"3 条执行中、2 条待确认"）

底部区域：**统一输入口**（语音 + 文字 + 图片）

- 在任何页面都可以立刻发新指令
- 不需要先退出当前详情

#### B. 任务流

所有已发出的指令和对应的执行状态，按分层 + 时间排列。

每条任务卡片显示：

- 当前状态（理解中 / 执行中 / 待确认 / 已完成 / 失败）
- AI 对指令的理解（比原始指令更有价值）
- 当前正在做什么 / 结果摘要
- 如果需要确认：确认 / 取消按钮 + 为什么需要拍板
- 如果已完成：结果卡片 + 后续操作（继续处理、转文档、设提醒）
- 快捷纠偏入口：每条卡片支持一句话修正，不必进详情页

管理者可以点击任何一条任务查看详细执行过程。

#### C. 任务详情

点进某条任务后，展示完整的执行过程，固定为 4 段：

1. **我理解的是**：大模型对指令的理解
2. **我准备怎么做**：Agent Teams 的执行计划
3. **执行进展**：子任务列表及各自状态
4. **最终结果 / 需要你决定**：结果卡片或确认请求

管理者可以在任务详情里：

- 追问（"这个再详细说说"）
- 补充（"还有一个条件忘了说"）
- 纠偏（"不是这个意思""重点不对"）
- 改判（"这个不对，改成这样"）
- 叫停（"先别做了"）

任务详情页支持快速切换到其他活跃任务，不必频繁返回首页。

### 7.3 确认机制

确认不只是"确认/取消"，还必须说明：

- 将执行什么动作
- 影响谁
- 为什么这一步需要你拍板

执行中如果发现新风险，系统暂停并再次请示，不会静默继续。

### 7.4 结果后续动作

结果卡片不只是"看完关闭"。每个结果带 1—3 个自然下一步：

- 继续处理
- 转文档（简报、邮件草稿、会议纪要）
- 设提醒（"明天再提醒我看这个"）
- 转任务（把结果中的事项落地为任务）

### 7.5 异常突出

管理者不需要被所有进展打扰，但一定要被异常打扰。以下情况做强提醒：

- 卡住超过预期时长
- 信息不足无法继续
- 多方案冲突待拍板
- 外部动作执行失败
- 长期关注项出现异常波动

### 7.6 不需要的界面

- 不需要首页仪表盘
- 不需要功能菜单
- 不需要设置页面（偏好通过对话设定）
- 不需要 Bot 市场
- 不需要独立的任务管理界面

## 八、指令示例

不限定指令类型。以下是真实场景中管理者可能发出的指令：

### 8.1 纯语音

- "帮我整理一下昨天会议讨论的三件事，列个清单出来"
- "催一下小王，上周交代的方案还没给我"
- "今天下午 3 点的会准备一下议程"
- "帮我想想这个季度还有哪些目标没完成"

### 8.2 语音 + 图片

- （拍白板照片）"把这上面的事项都整理成任务"
- （截聊天记录）"这个客户反馈的问题帮我跟进一下"
- （拍合同照片）"看看这个合同有没有问题"

### 8.3 纯图片

- （发一张报表截图）系统识别后主动问："需要我分析这份数据吗？"

### 8.4 纯文字

- "把上次那个方案再改改，把预算部分详细拆一下"
- "本周五之前把所有延期事项处理完"

### 8.5 追问和补充

- "刚才那个事，加一个条件：预算不能超过 10 万"
- "上午说的那个客户的事，现在进展怎么样了"
- "这个方案不行，换个思路"

## 九、执行能力与工具体系

### 9.1 第一版内置工具

不依赖外部系统，产品自身提供：

- **任务管理**：创建任务、更新状态、设定截止时间、标记完成
- **信息整理**：整理文本、提取图片内容、生成摘要、制作清单
- **文档生成**：写方案、写简报、写邮件草稿、写会议议程
- **提醒调度**：设定提醒、定时检查、周期性汇报
- **记忆存取**：存储和检索管理者的历史指令、偏好、关键信息

### 9.2 后续可扩展工具

接入外部系统后增加的能力：

- **通知工具**：发消息给团队成员（接入 IM）
- **日历工具**：查看 / 创建 / 修改日程（接入日历）
- **邮件工具**：发送 / 读取邮件（接入邮箱）
- **数据工具**：查询业务数据（接入数据库 / BI）
- **流程工具**：发起审批、触发流程（接入 OA）

### 9.3 关键原则

工具是可插拔的，产品的核心体验不依赖任何外部系统。

### 9.4 执行边界

- **默认可执行**：读取、整理、分析、生成、提醒等不产生外部影响的动作
- **需要确认**：发送通知、修改数据、触发流程等产生外部影响的动作
- **拒绝执行**：明显违规、超出权限、可能造成不可逆损害的动作

边界判断由大模型根据管理者的角色和规则自行决定，不需要提前硬编码每种情况。

## 十、上下文与记忆

### 10.1 管理者画像

管理者首次使用时，通过对话完成初始化：

> "你好，先跟我说说你是做什么的，平时管哪些事？"

管理者用自然语言描述即可，大模型自动提取：

- 角色
- 职责范围
- 管理对象
- 常见工作内容
- 偏好和习惯

随着使用，画像持续丰富，不需要填表或配置。

### 10.2 任务记忆

所有指令和执行记录持久化存储。Agent Teams 可以随时检索历史：

- "上次那个事" → 系统知道指的是哪件事
- "跟之前一样处理" → 系统知道之前是怎么处理的
- "这个月我一共交代了多少件事" → 系统能统计

### 10.3 规则记忆

管理者可以通过对话设定持续生效的规则：

- "以后这类事不用问我，直接做"
- "涉及预算的事一律先给我看"
- "每天早上帮我列一下今天要关注的事"

规则由大模型记住并持续执行，不需要进设置页面配置。

## 十一、关键体验要求

### 11.1 输入必须零门槛

- 拿起手机，按住说话，松开就行
- 不需要选类别、选 Bot、选模板
- 想发图片就发图片，想说话就说话

### 11.2 反馈必须即时

- 发出指令后 1 秒内看到"已收到"
- 大模型理解后立即告诉管理者"我理解你要做的是……"
- 如果理解有误，管理者可以立即纠正

### 11.3 执行必须透明

- 管理者随时能看到 Agent Teams 在做什么
- 不是黑箱，而是"我能看见我的团队在干活"
- 但不需要管理者逐步审批每一步

### 11.4 确认必须精准

- 只在真正需要管理者拍板的时候才打断他
- 不要什么都问，也不要什么都不问
- 大模型根据管理者的规则和指令内容自行判断

### 11.5 结果必须可用

- 最终输出不是一段聊天文字
- 而是可操作的结果：清单、文档、方案、提醒、任务记录
- 管理者可以直接用、直接转发、直接确认

## 十二、与现有产品的本质区别

| 维度 | 现有 AI 助手 | 现有企业软件 | 本产品 |
|------|------------|------------|--------|
| 输入 | 文字对话 | 表单 / 菜单 | 语音 / 文字 / 图片，任意形式 |
| 能力 | 回答问题 | 预定义功能 | 理解任意指令并执行 |
| 执行 | 不执行 | 人工操作 | Agent Teams 自动执行 |
| 持续性 | 单次对话 | 人推才动 | 持续推进 |
| 控制 | 无 | 流程审批 | 关键节点确认 |
| 扩展性 | 模型能力 | 开发新功能 | 增加工具即增加能力 |

## 十三、版本规划

### V0.1 — 验证核心闭环

- 移动端基础界面：首页（输入区 + 任务流）
- 支持语音和文字输入
- 大模型理解指令
- 单 Agent 执行简单任务（整理信息、生成文档、创建任务）
- 返回结果卡片
- 失败处理（哪步失败、原因、建议补救、重试）

**验证点**：管理者发一句话，系统能正确理解并执行，结果可用。

### V0.2 — 多任务 + 确认 + 多模态

- 多任务并行执行，局部阻塞
- 首页分层展示（待拍板 → 执行中 → 已完成）
- 支持图片输入（拍照、截图、混合输入）
- 多 Agent 并行执行复杂任务
- 确认机制（风险分级、确认理由、确认/取消）
- 追问 / 补充 / 纠偏 / 叫停
- 任务归属识别（新任务 vs 旧任务补充）
- 任务详情页：查看完整执行过程

**验证点**：连续发 3 条指令 → 并行推进 → 其中一条需确认 → 其余不受阻塞。

### V0.3 — 记忆 + 持续推进

- 管理者画像建模
- 历史指令和任务记忆
- 规则记忆（从对话设定、持续生效）
- 历史复用（"上次那个""按之前那个风格"）
- 持续关注（"帮我盯着"类长期任务）
- 阶段汇报（里程碑、异常时汇报，普通进展不打扰）
- 晨间简报 / 晚间复盘
- 任务依赖和批量拍板
- 结果后续动作（继续处理、转文档、设提醒、转任务）

**验证点**："帮我盯着"类指令持续运行 → 发现异常主动汇报 → 晨间简报可用。

### V0.4 — 外部工具 + 协作

- 接入通知（IM）、日历、邮件
- 协作闭环（下发任务给人、回收反馈、自动催办）
- 完整审计（操作留痕、追责链路）
- 打扰预算（控制主动提醒频次）

**验证点**：一条指令 → 自动发消息给负责人 → 回收反馈 → 汇总。

### V1.0 — 完整的知知

- 管理风格学习（输出风格、确认习惯、关注重点）
- 关系记忆（人、项目、客户、区域）
- 冲突检测（目标冲突、规则冲突、重复指令）
- 数据/BI 接入
- 结果可信度标注

## 十四、成功指标

### 14.1 核心指标

- 日均指令数：管理者每天发多少条指令
- 指令完成率：发出的指令有多少被成功执行
- 二次编辑率：管理者需要纠正或补充的比例（越低越好）
- 次日留存率：用了第一天后第二天还用的比例

### 14.2 体验指标

- 指令到反馈时长
- 大模型理解准确率
- 确认节点命中率（该问的问了，不该问的没问）
- 管理者对结果满意度

### 14.3 终极指标

- 管理者是否觉得"少了它不行"
- 管理者是否把它当成自己的参谋团队而不是一个软件

## 十五、风险与挑战

### 15.1 大模型理解准确率

管理者的指令经常模糊、省略、指代不清。大模型必须能结合上下文正确理解。

应对：持续优化 prompt、丰富管理者上下文、允许管理者快速纠正。

### 15.2 Agent Teams 执行可靠性

Agent 可能执行出错、超时、死循环。

应对：设定执行超时、错误重试、失败降级、管理者可随时叫停。

### 15.3 长时间任务的持续性

管理者说"帮我盯着"，可能要盯几天。

应对：定时任务机制、状态持久化、定期汇报。

### 15.4 安全与权限

Agent Teams 不能做超出管理者权限的事。

应对：大模型根据角色和规则判断边界，高风险动作必须确认，所有操作可审计。

## 十六、产品壁垒

壁垒不在"模型会不会聊天"，而在：

- 对管理语义的理解深度
- 管理者上下文的持续积累
- Agent Teams 工具生态的丰富度
- 确认与控制机制的精准度
- 对个体管理者风格的学习和适配

用得越久，系统越懂这个管理者，迁移成本越高。

## 十七、一句话总结

**这个产品只做一件事：让管理者用最自然的方式发出任何指令，然后由 AI Agent Team 去执行，管理者只管发令和拍板。**
